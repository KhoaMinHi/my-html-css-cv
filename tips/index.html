git, commit, git cherry-pick -n, git commit --amend --no-edit

Khi làm việc với team, những lúc khách hàng yêu cầu cần release gấp 1 feature. Mà feature này đã được làm đầu tuần, cuối tuần khách hàng mới báo. Khi đó commit của bạn ở giữa commit của team.

Cách thủ công nhất là bạn phải đi copy từng dòng code mà mình đã code trước đó ở đầu tuần. Với sớ lượng code ít hoặc bạn nhớ mình code chỗ nào thì oki; còn không nhớ hoặc commit code nhiều lần, xen kẽ code thành viện khác trong team thì oh my god con lạy chúa. Lúc đó rất dễ sót code.

Cách thứ 2 là dùng git, đặc biệt là git cherry-pick. Nó giúp bạn lấy từng commit (dựa vào hash id) để apply vào nhánh bạn đang đứng.

*Node: để làm tốt việc mình đã có commit nào liên quan tới task jira (reference jira link) thì bạn phải có thói quên commit với comment để search

git switch branch_name # chọn nhánh cần apply commit vào (nhánh master)
git status # kiểm tra
git branch feat_temp # tạo một nhánh tạm thời, không nên apply commit thẳng vào nhánh master
git checkout feat_temp # bắt đầu thực hiện trên nhánh này
git cherry-pick -n hashid_oldest1 # -n là option để không commit, tức là nó sẽ apply code và code ở trạng thái stage (thường là màu xanh, sau working tree). Dùng option này để khi cherry-pick có thể bị conflict code. Và chúng ta vẫn append được vào 1 commit duy nhất bằng lệnh git commit -amend --no-edit.
=> nếu có conflict thì giải quyết conflict và chạy git add . (hoặc git add file_path). Sau đó commit lại (git commit -m "comment").
git cherry-pick -n hashid_oldest2 # commit cũ thứ 2
git commit -amend --no-edit
git cherry-pick -n hashid_oldest3
git commit -amend --no-edit
...
 => push branch mới này lên origin (gitlab.com, github.com,...) git push --set-upstream origin khoa
merge nhánh này vào master
 merge nhánh này vào nhánh chung khác của team (dev, test,...)
